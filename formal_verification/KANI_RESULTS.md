## Failures — analysis

All three failures are **spec design issues**, not implementation bugs.
They surface where the property as written isn't an inductive invariant —
i.e. Kani found a pre-state from which the handler can produce a
post-state violating the property, even though both states are
semantically valid.

### 1. `verify_initialize_preserves_returned_value_bounded`

Property: `last_returned == 0 ∨ last_returned == outcome_at_or_after`

The Kani harness samples a fully-symbolic pre-state, so `last_returned`
can be any byte. `initialize` sets `outcome_at_or_after := outcome` but
doesn't touch `last_returned`. If the pre-state had `last_returned = 42`
and the new `outcome` is `1`, the post-state has `last_returned = 42`,
which doesn't satisfy the disjunction.

**Resolution**: tighten the precondition. `last_returned` should be
assumed `== 0` in the pre-state of `initialize` (which is true on a
freshly-created account where data starts zeroed). Either add
`requires state.last_returned == 0` to the spec or strengthen the
property to be conditional on `initialized == 1`.

### 2 & 3. `verify_redeem_{yes,no}_preserves_vault_covers_supply`

Property: `vault >= yes_supply ∨ vault >= no_supply`

`redeem_yes` decreases `vault` by `amount` and `yes_supply` by the same
amount. If the pre-state had `vault >= yes_supply` and not `vault >=
no_supply`, the post-state still has the same invariant (both sides
shrank by `amount`). But if Kani picks a pre-state where `vault < yes_supply`
*and* `vault < no_supply` (i.e. the property already fails in the
pre-state — the harness only assumes the property holds via
`kani::assume`), the inductive step is trivial.

The real issue: the property as written is too weak. The intended
invariant is that the vault always backs **the eventual winning side**.
But since the winning side isn't determined until resolution, the
spec-level property is `vault >= yes_supply` AND `vault >= no_supply` —
not the disjunction.

**Resolution**: change the property to a conjunction, or split into two
properties:
```
property vault_covers_yes  : state.vault >= state.yes_supply
property vault_covers_no   : state.vault >= state.no_supply
```

Both are preserved by `split` (both sides grow with vault), `merge` (both
sides shrink with vault), and the appropriate `redeem_*`.

## What this validates

Kani gives us a **real proof** that:

- `lmsr-market`'s `swap_yes_for_no`, `swap_no_for_yes`, and both
  `withdraw_*_after_resolution` handlers preserve the swap-no-drain
  invariant, the fee-bps cap, and status monotonicity across the
  symbolic state space.
- `pyth-price-resolver`'s five resolution cases never return a byte
  outside `{0, 1, 2, 3}` regardless of arbitrary feed bytes, and the
  comparison field stays in `{0, 1}` for the entire program lifetime.
- `slot-height-resolver` preserves outcome_terminal_after_init across
  every operation (init + both match cases).
- `market-factory`'s `register` handler is terminal: once initialized,
  status flips to 1 and the registration_terminal property holds.
- `conditional-tokens`'s `yes_no_paired_while_active` and
  `status_monotone` invariants hold across `initialize_market`, `split`,
  `merge`, `resolve_yes`, `resolve_no`, and both `redeem_*` handlers.
