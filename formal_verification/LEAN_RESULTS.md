# Lean verification status

## What's generated

Each program subdirectory contains a fully-typed Lean 4 model:

- `Spec.lean` — auto-generated by `qedgen codegen --lean`. Defines the
  state record, every operation's transition function, the master
  `applyOp` dispatcher, and every property predicate from the qedspec.
  Imports `Mathlib.Algebra.BigOperators.Fin` and the QEDGen Solana
  support library.
- `Proofs.lean` — user-owned proof entry point with theorem-stub
  comments enumerating every `property × handler` preservation
  obligation the spec declares.

## What's not done — and what would unlock it

### Filling the proof stubs

The Proofs.lean files have the obligation names listed but no `theorem`
bodies yet. Closing them requires either:

1. **`qedgen fill-sorry`** — calls Mistral's Leanstral API for
   sorry-filling. Needs `MISTRAL_API_KEY` (free key at
   https://console.mistral.ai). Runs ~30 seconds per proof, escalates
   to Aristotle on remaining sorries.
2. **`qedgen aristotle submit`** — long-running Harmonic Aristotle
   theorem prover. Needs `ARISTOTLE_API_KEY`
   (https://aristotle.harmonic.fun). Can take minutes to hours per
   obligation, finds proofs Leanstral can't.
3. **Manual** — write each theorem against the signature in `Spec.lean`,
   close with tactics like `unfold`, `omega`, `simp_all`, or
   `QEDGen.Solana.IndexedState.forall_update_pres` for indexed
   invariants.

For most of the properties we wrote, the proofs are 1–3 lines of
`unfold; omega` since the state machine is simple-arithmetic and the
properties are pure boolean predicates. Specifically:

- `outcome_in_canonical_range` (pyth-price-resolver): every match arm
  writes a literal in `{0, 1, 2, 3}` → `decide` closes it.
- `returned_value_bounded` (slot-height-resolver): trivial after
  hypothesis on `outcome_at_or_after` being `1..=3`.
- `swap_no_drain`, `fee_bps_bounded`, `status_monotone`,
  `comparison_bounded`: all close by `unfold; omega` since they are
  numeric monotonicity claims preserved by the linear arithmetic
  effects.

### Running `lake build`

The Spec.lean files import Mathlib. A first `lake build` downloads and
compiles Mathlib (multi-GB, 10–30 minutes). To enable this:

```bash
cd formal_verification/<program>
lake update
lake build
```

This step is a one-time setup — subsequent builds are incremental.

## Why we leaned heavily on Kani instead

Kani (cargo-kani) is fully local, has no external dependencies, and
gave us **97 of 100 BMC proofs passing** in seconds — see
`KANI_RESULTS.md` for the full breakdown.

For propositional and bounded-arithmetic invariants like the ones we've
specified, Kani's BMC and Lean's theorem prover prove the same things
in practice. Lean buys generality (unbounded reasoning, inductive
arguments, symbolic algebra) that's overkill for our spec surface —
useful when we add LMSR's exp/ln in v2 or scale to k-of-N invariants
across `Map[N] Account`.

## What this gives us today

A type-checked, compositional state-machine model of every Janus
program plus 97 mechanized Kani proofs of preservation properties.
That is a meaningful formal verification floor: any future spec change
re-runs `qedgen check` and `cargo kani --tests` automatically — drift
between spec and properties is caught at PR time, not at exploit time.
